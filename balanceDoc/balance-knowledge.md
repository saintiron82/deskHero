## 0. 공식 통일 시스템 (CRITICAL)

### Single Source of Truth 아키텍처

```
config/StatFormulas.json  ← 유일한 공식 정의 파일
        ↓
tools/generate_stat_code.py (코드 생성기)
        ↓
    ┌───────┴───────┐
    ↓               ↓
Python 코드        C# 코드
(밸런스 도구)      (실제 게임)
    ↓               ↓
    └─── 동일 공식 ──┘
```

### 왜 중요한가?

- **밸런스 예측 = 실제 게임 결과** 보장
- 공식이 달라지면 모든 분석이 무의미해짐
- 양쪽(Python/C#)이 항상 동일한 계산 수행

### 공식 변경 절차

```bash
# 1. config/StatFormulas.json 수정 (여기만!)
# 2. 코드 생성기 실행
python tools/generate_stat_code.py

# 3. 검증
python tools/test_stat_formulas.py
dotnet build
```

## 1. 핵심 공식

### 1.1 업그레이드 비용 공식
```
upgrade_cost = base_cost × (1 + level × growth_rate) × multiplier^(level / softcap_interval)
```

| 변수 | 설명 | 일반적인 범위 |
|------|------|--------------|
| `base_cost` | 기본 비용 (Lv1 기준) | 1~100 |
| `growth_rate` | 레벨당 선형 증가율 | 0.1~1.0 |
| `multiplier` | 급등 배수 (지수 증가) | 1.2~2.0 |
| `softcap_interval` | 급등 적용 주기 | 5~20 |
| `level` | 현재 레벨 | 1~100+ |

**예시:** base_cost=10, growth_rate=0.5, multiplier=1.5, softcap_interval=10, level=20
```
cost = 10 × (1 + 20 × 0.5) × 1.5^(20/10)
     = 10 × 11 × 2.25
     = 247.5 → 247
```

### 1.2 몬스터 HP 공식
```
monster_hp = BASE_HP × HP_GROWTH^stage
보스 HP = monster_hp × BOSS_HP_MULTI
```

| 상수 | 값 | 설명 |
|------|-----|------|
| `BASE_HP` | 100 | 1스테이지 HP |
| `HP_GROWTH` | 1.2 | 스테이지당 HP 증가율 |
| `BOSS_HP_MULTI` | 5.0 | 보스 HP 배수 |
| `BOSS_INTERVAL` | 10 | 보스 등장 주기 |

**예시 HP 테이블:**
| 스테이지 | 일반 HP | 보스 HP |
|----------|---------|---------|
| 1 | 120 | - |
| 10 | 619 | 3,095 |
| 20 | 3,834 | 19,170 |
| 30 | 23,738 | 118,690 |
| 50 | 910,044 | 4,550,220 |

### 1.3 데미지 계산 공식
```
damage = (base_power + base_attack) × (1 + attack_percent%) × crit_expected × multi_expected × combo_multi
```

**세부 계산:**
```
1. 기본 데미지 = base_power
2. 가산 데미지 = base_power + base_attack
3. 배수 데미지 = step2 × (1 + attack_percent / 100)
4. 크리티컬 기대값 = 1 + crit_chance × (crit_multi - 1)
5. 멀티히트 기대값 = 1 + multi_hit_chance / 100
6. 콤보 배율 = (1 + combo_damage / 100) × 2^combo_stack
```

| 기본 상수 | 값 | 설명 |
|----------|-----|------|
| `BASE_CRIT_CHANCE` | 0.1 | 기본 크리티컬 확률 (10%) |
| `BASE_CRIT_MULTIPLIER` | 2.0 | 기본 크리티컬 배수 |
| `BASE_TIME_LIMIT` | 30 | 스테이지 제한 시간 (초) |
| `MAX_COMBO_STACK` | 3 | 최대 콤보 스택 |
| `COMBO_DURATION` | 3.0 | 콤보 유지 시간 (초) |
| `BASE_HP` | 100 | 1스테이지 몬스터 HP |
| `HP_GROWTH` | 1.2 | 스테이지당 HP 증가율 |
| `BOSS_INTERVAL` | 10 | 보스 등장 주기 |
| `BOSS_HP_MULTI` | 5.0 | 보스 HP 배수 |
| `BASE_GOLD_MULTI` | 1.5 | 기본 골드 배수 |
| `GOLD_TO_CRYSTAL_RATE` | 1000 | 골드→크리스탈 환율 |

### 1.4 필요 CPS 공식
```
required_CPS = (monster_HP / damage) / time_limit
```

**의미:** 해당 스테이지를 30초 내에 클리어하려면 초당 몇 번 클릭해야 하는가

### 1.5 골드 획득 공식
```
gold_earned = (base_gold + gold_flat + gold_flat_perm) × (1 + gold_multi + gold_multi_perm)
base_gold = stage × 1.5
```

### 1.6 크리스탈 획득 공식

**수입원 1: 스테이지 클리어 보상 (초반 부스터)**
```
crystal_per_stage = 1  (매 스테이지 클리어 시)
```

**수입원 2: 보스 드롭 (메인 수입원)**
```
drop_chance = min(base_chance + crystal_multi / 100, max_chance)
crystal_amount = base_crystal + (boss_level × crystal_per_level) × (1 ± variance)
```

| 상수 | 값 | 설명 |
|------|-----|------|
| `base_chance` | 0.5 (50%) | 기본 드롭 확률 |
| `max_chance` | 0.95 (95%) | 최대 드롭 확률 |
| `base_crystal` | 5 | 기본 드롭량 |
| `crystal_per_level` | 1 | 보스 레벨당 추가량 |
| `variance` | ±20% | 분산 |

**크리스탈 수입 비교 (스테이지별):**
| 도달 스테이지 | 스테이지 보상 | 보스 보상 (추정) | 비중 |
|--------------|--------------|-----------------|------|
| Stage 1-10 | 10💎 | ~5💎 | 스테이지 > 보스 |
| Stage 20-30 | 10💎 | ~15💎 | 비슷 |
| Stage 40-50 | 10💎 | ~30💎 | 보스 > 스테이지 |

**설계 의도:**
- 스테이지 보상: 초반 부스터 (고정 1💎)
- 보스 보상: 후반 메인 수입원 (레벨 비례)
- 후반으로 갈수록 보스 드롭 비중 증가

## 2. 밸런스 판단 기준 (CRITICAL)

### 2.1 핵심 지표: 필요 CPS

> **가장 중요한 질문: "클리어하려면 초당 몇 번 입력해야 하는가?"**

시간 제한은 time_extend 등으로 조절 가능하므로, **필요 CPS**가 진정한 난이도 지표입니다.

### 2.2 필요 CPS 계산 공식

```
필요_CPS = HP / (파워 × 콤보배율 × 제한시간)
```

**예시 (Lv20 보스, 콤보 3스택, 30초):**
```
필요_CPS = 14,695 / (40 × 8 × 30)
         = 14,695 / 9,600
         = 1.53 CPS  ← 초당 1.53회 입력이면 클리어
```

### 2.3 CPS 난이도 판정 기준

| 필요 CPS | 판정 | 설명 |
|----------|------|------|
| **< 3** | ✅ 매우 여유 | 캐주얼 플레이 가능 |
| **3~5** | ✅ 여유 | 일반 플레이어 클리어 가능 |
| **5~8** | ⚠️ 도전적 | 숙련 필요 또는 업글 권장 |
| **8~12** | ⚠️ 어려움 | 업그레이드 또는 time_extend 필요 |
| **12~15** | ❌ 극한 | 최고 숙련도 필요, 업글 권장 |
| **> 15** | ❌ 불가 | **입력 한계 초과** - 업그레이드 필수 |

**인간 입력 한계:** 최대 15 CPS (극소수 도달 가능)
**일반인 기준:** 평균 5~7 CPS, 숙련자 10~12 CPS

### 2.4 레벨별 필요 CPS 테이블 (콤보 3스택, 30초)

| Lv | 보스 HP | 파워 | 데미지/입력 | **필요 CPS** | 판정 |
|----|---------|------|-------------|--------------|------|
| 1 | 460 | 2 | 16 | **0.96** | ✅ 매우 여유 |
| 10 | 2,370 | 20 | 160 | **0.49** | ✅ 매우 여유 |
| 20 | 14,695 | 40 | 320 | **1.53** | ✅ 매우 여유 |
| 30 | 90,990 | 60 | 480 | **6.32** | ⚠️ 도전적 |
| 40 | 563,410 | 80 | 640 | **29.3** | ❌ 업글 필수 |
| 50 | 3,488,500 | 100 | 800 | **145.4** | ❌ 업글 필수 |

### 2.5 단계별 기대값

| 단계 | 레벨 | 필요 CPS (목표) | 필요 조건 |
|------|------|-----------------|-----------|
| 튜토리얼 | 1~10 | < 1 | 콤보 유지 |
| 초반 | 10~20 | < 2 | 콤보 3스택 |
| 중반 | 20~30 | < 7 | 콤보 + 인게임업글 |
| 후반 | 30~40 | < 10 (업글) | 영구업글 투자 |
| 엔드게임 | 40+ | 업글로 해결 | 상당한 영구업글 |

### 2.6 밸런스 품질 지표

**좋은 밸런스:**
- Lv1~20: 필요 CPS < 3 (누구나 클리어)
- Lv20~30: 필요 CPS 3~8 (도전적이지만 가능)
- Lv30+: 업그레이드로 필요 CPS를 낮춰서 해결

**나쁜 밸런스:**
- 초반부터 필요 CPS > 5 → 진입장벽 과다
- 업글해도 필요 CPS 감소 미미 → 성장감 부재
- 콤보 없이도 필요 CPS < 3 → 핵심 메커니즘 무의미

## 3. 파라미터별 영향 분석

### 3.1 base_cost (초기 비용)
```
영향: 전체 비용 스케일 조절

↑ 높이면:
  - 전체 비용 곡선이 위로 이동
  - 초반 진입장벽 상승
  - 후반 영향은 상대적으로 감소 (multiplier가 지배적)

↓ 낮추면:
  - 초반 접근성 향상
  - 상대적으로 저렴한 초반 투자

조정 팁: 초반 밸런스 조절에 사용
```

### 3.2 growth_rate (증가율)
```
영향: 선형적 비용 증가 속도

↑ 높이면:
  - 레벨이 오를수록 비용 빠르게 증가
  - 후반 비용 증가 (선형)
  - multiplier와 복합 작용

↓ 낮추면:
  - 레벨 간 비용 차이 감소
  - 꾸준한 성장 가능

조정 팁: 0.3~0.7 범위 권장
```

### 3.3 multiplier (급등 배수)
```
영향: 지수적 비용 폭발

↑ 높이면 (1.5 이상):
  - 후반 비용 기하급수적 증가
  - softcap 구간마다 급등
  - 하드 캡 효과 (사실상 최대 레벨 제한)

↓ 낮추면 (1.3 이하):
  - 선형에 가까운 비용 증가
  - 무한 성장 가능

조정 팁:
  - 강한 제한: 1.8~2.0
  - 보통: 1.4~1.6
  - 약한 제한: 1.2~1.3
```

### 3.4 softcap_interval (급등 주기)
```
영향: 급등 발생 빈도

↑ 높이면 (20 이상):
  - 급등 간격이 넓어짐
  - 완만한 비용 곡선
  - 더 많은 레벨까지 접근 가능

↓ 낮추면 (5~10):
  - 자주 급등 발생
  - 가파른 비용 곡선
  - 빠른 하드 캡

조정 팁: CPS 균형도가 낮을 때 증가 권장
```

### 3.5 effect_per_level (레벨당 효과)
```
영향: 투자 보상

↑ 높이면:
  - 레벨당 효과 증가
  - 투자 효율 상승
  - CPS 더 많이 감소

↓ 낮추면:
  - 레벨당 효과 감소
  - 고레벨까지 필요

조정 팁: 비용과 함께 조절하여 효율 균형 유지
```

## 4. 밸런스 조정 가이드라인

### 4.1 문제별 해결책

#### CPS 균형도가 낮음 (효율 편차 높음)
```
원인: 특정 레벨 구간의 효율이 다른 구간과 크게 다름

해결책:
1. softcap_interval 증가 (급등 완화)
2. growth_rate 감소 (선형 증가 완화)
3. effect_per_level 조정 (보상 재분배)
```

#### 초반만 효율 좋음
```
원인: multiplier가 너무 높거나 softcap_interval이 너무 낮음

해결책:
1. multiplier 감소 (1.3~1.5 권장)
2. softcap_interval 증가
3. 후반 effect_per_level 보정 고려
```

#### 후반 접근 불가 (CPS > 15)
```
원인: 스탯 효과가 HP 증가를 따라가지 못함

해결책:
1. effect_per_level 증가
2. multiplier 감소 (비용 낮춤)
3. 여러 스탯의 시너지 확인
```

#### 비용이 너무 저렴함
```
원인: base_cost, growth_rate, multiplier 모두 낮음

해결책:
1. base_cost 증가 (초반 조절)
2. growth_rate 증가 (중반 조절)
3. multiplier 증가 (후반 조절)
```

### 4.2 스탯별 권장 범위

| 스탯 | base_cost | growth_rate | multiplier | softcap | effect |
|------|-----------|-------------|------------|---------|--------|
| base_attack | 5~20 | 0.3~0.6 | 1.3~1.6 | 10~20 | 1~5 |
| attack_percent | 10~30 | 0.4~0.7 | 1.4~1.7 | 10~15 | 1~3 |
| crit_chance | 10~25 | 0.3~0.5 | 1.3~1.5 | 10~15 | 0.5~2 |
| crit_damage | 15~35 | 0.4~0.6 | 1.4~1.6 | 10~15 | 0.1~0.3 |
| multi_hit | 20~40 | 0.5~0.8 | 1.5~1.8 | 8~12 | 1~3 |
| time_extend | 30~50 | 0.6~0.9 | 1.6~2.0 | 5~10 | 0.5~1 |

### 4.3 조정 우선순위

1. **softcap_interval** - 가장 직관적, 곡선 형태 조절
2. **multiplier** - 후반 난이도 조절
3. **growth_rate** - 중반 난이도 조절
4. **effect_per_level** - 보상 스케일 조절
5. **base_cost** - 초반 난이도 조절

## 실제 측정 데이터
